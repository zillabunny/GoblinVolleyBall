## Devil's Advocate Review: Failure Modes Analysis

Based on my thorough review of the final plan, here are the NEW failure modes and incomplete fixes I found:

### **CRITICAL ISSUE #4: Server Physics Cannot Receive hitPressed**

**The Problem:** The client sends `{ type: 'input', payload: { keys: { left, right, jump, hit } } }` (line 1002-1003), but the server expects `inputs: [{ left, right, jump, hit, hitPressed }, ...]` (line 846). The server has no way to compute `hitPressed` because it only receives the current key state, not the previous state.

The comment at line 846 says the server accepts `hitPressed`, but the client's `sendInput()` method at line 1000-1003 never sends it. The server would need to track `_hitPrev` per player to reconstruct `hitPressed`, which is not shown in the plan.

**Impact:** Every hit on the server will fail because `tryHit()` checks `if (!input.hitPressed) return;` at line 474. Server-side hits will never execute in multiplayer mode.

---

### **CRITICAL ISSUE #2: AI hitPressed Set Every Frame in Range**

**The Problem:** At line 698, the AI does:
```js
if (dist <= AI_PARAMS.hitThreshold && dist >= HIT_MIN_DIST) {
  input.hitPressed = true;
}
```

This sets `hitPressed = true` EVERY frame the AI is in range. The AI doesn't create a fresh input object each frame — it returns the same input object (or a new one with `hitPressed` left as `true`).

The issue: Unlike the human player's `InputState`, which uses `hitPressed` getter to compute `this.hit && !this._hitPrev`, the AI controller directly sets `input.hitPressed = true`. There's no `_hitPrev` tracking. So `hitPressed` will be true for multiple consecutive frames, and `tryHit()` will execute multiple times on the ball while the AI is in range.

**Impact:** The AI will hit the ball multiple times per approach instead of once per serve/rally. This is physically impossible and will corrupt game state.

---

### **ISSUE #6: Touch Input Won't Trigger Hits**

**The Problem:** At lines 762-763:
```js
this[key] = true;
if (key === 'hit') this.hit = true;
```

The touch handler sets `this.hit = true`, but never sets `hitPressed`. The touch button will work only if the touch triggers the *next* frame where `hitPressed` is computed by the getter at line 387:
```js
get hitPressed() {
  return this.hit && !this._hitPrev;
}
```

This will work once (on the frame after touch), but only if:
1. The previous frame had `_hitPrev = false`
2. `tick()` hasn't been called yet this frame
3. The touch happens at the *right* time relative to the input.tick() call

**Actual problem:** The touch sets `this.hit = true` immediately. If `hitPressed` is already read this frame (before touch), the getter still sees `_hitPrev = false` from last frame, so `hitPressed = true` and works. But on the *next* frame, `tick()` sets `_hitPrev = true`, so `hitPressed = false` unless the player holds the button (which doesn't make sense for touch). Touch hits are one-shot by accident, only by timing.

For a mobile player, this is fragile — if they touch quickly, they might miss the one-frame window if the touch arrives after `game.update()` reads `hitPressed` but before `tick()` is called.

**Impact:** Touch hits may fail to register or require weird timing. The plan doesn't document that touch must arrive between `hitPressed` read and `tick()` call.

### **ISSUE #1: _lastServeSeq Logic Is Fragile But Likely Works**

**The Problem:** At lines 520-523:
```js
if (state.touchSeq > state._lastServeSeq) {
  state._lastServeSeq = state.touchSeq;
  state.phase = 'playing';
}
```

The initialization at line 586 sets `state._lastServeSeq = -1`. The first hit increments `touchSeq` to 1 at line 494, so `1 > -1` is true, and the phase transitions. This works.

However, the fragility: `touchSeq` is a global counter that increments on every hit. If somehow a hit happens twice per frame (impossible in normal physics, but see Issue #2 about the AI), or if `touchSeq` wraps around (at 2^53 - 1 after billions of hits), this logic could fail. But in practice, with one hit per serve and ~60fps gameplay, this will work for years.

**Verdict:** Not a critical bug, but relies on no duplicate hits per frame (which the AI issue violates).

**The Problem:** The InputState `tick()` must be called *after* `hitPressed` is read. At line 722:

This is called AFTER `tryHit()` at lines 718-719, so the order is:

**Verdict:** Works by design, but fragile. No mechanism prevents future bugs.
