The 20Hz tick rate meant 50ms between state updates — players saw choppy movement and missed ball interactions. Bumping to 60Hz matched the client's render rate, giving smooth, responsive gameplay. Trade-off: more bandwidth, but Goblin Volleyball's state is tiny.

Netcode Architecture

Client

  • Capture input (keys/touch)
  • Client-side prediction
  • Apply local player physics
  • Send input on change only
  • Receive server snapshots
  • Reconcile position (50px threshold)
  • Render: server state + local prediction
  • Detect & play sound effects
INPUT
WebSocket
STATE

Server

  • Authoritative game simulation
  • Full physics (ball, players, net)
  • Scoring & phase management
  • 60 Hz fixed-step update loop
  • Broadcast state snapshots
  • Room-based matchmaking
  • Apply received inputs to players
  • Fly.io (Docker container)
● Input packets: {left, right, jump, hit} — sent only when changed
● State snapshots: full game state at 60Hz — positions, velocities, phase, score

Lessons Learned

1. Don't fight the server. The first online build had the client running full physics alongside the server. Both simulated the ball, scoring, and opponent — and they disagreed constantly. The fix: strip the client to movement-only and let the server own everything else.

2. Interpolation isn't always the answer. Classic netcode advice says to interpolate between snapshots. We tried it, but at 60Hz the snapshots arrive every 16ms — already one render frame. Interpolation added complexity and latency for zero visual benefit. Removed it.

3. Send less, not more. Sending input every frame (60/s) was wasteful. Most frames, nothing changes. Switching to on-change-only cut bandwidth by ~95% with no gameplay impact.

4. Trust prediction, but verify. Pure prediction without reconciliation drifts. Pure server authority without prediction feels laggy. The sweet spot: predict locally, but hard-snap back if the server says you're more than 50px off.
